# Roadmap

Текущая версия: **MVP+** (v0.5).

## Что уже есть

### Клиент
- FSM-диалог: устройство → модель → проблема (текст/голос) → фото → телефон → подтверждение
- Кнопка отправки контакта `request_contact` + ручной ввод с нормализацией в E.164
- Кнопка «Изменить» на шаге подтверждения
- `/myorders` — список своих заявок
- `/cancel`, `/help` — глобальные команды
- После создания заявки — все текстовые/фото сообщения автоматически уходят мастеру

### Мастер
- Inline-кнопки `[💬 Ответить][✅ В работе][🏁 Закрыть]` под каждым уведомлением
- Свайп-реплай на уведомление = ответ клиенту (без нажатия кнопок)
- В форум-режиме: каждое сообщение в топике = ответ клиенту этой заявки
- `/orders` — список открытых заявок
- `/done` — выйти из активного диалога

### Архитектура
- Слои: `handlers/`, `services/`, `database/repositories/`, `keyboards/`, `middlewares/`, `domain/`, `utils/`
- Миграции версионированы через `PRAGMA user_version` (5 миграций)
- Полный аудит: `order_status_history` + `order_messages` + `order_photos`
- Throttling, idempotency на confirm, retry с exponential backoff
- Pydantic Settings, startup-проверка чата мастера
- Двойной режим: форум-супергруппа (топик на заявку) ↔ личный чат (reply_to)

### Что в БД
- `orders` — основная таблица (с `device_model`, `topic_id`, `notification_message_id`, `updated_at`)
- `order_status_history` — кто, когда, с какого на какой статус
- `order_messages` — переписка клиент↔мастер
- `order_photos` — фото к заявке (one-to-many)

## Что я бы выбрал делать следующим

**Сейчас → v0.6 (форум-режим)** + **v0.7 (шаблоны)** + **#13 (Docker)**.

Это закроет 80% реальной работы мастера: настоящие изолированные окна по заявкам, быстрые ответы, и удобный деплой на VPS.

**Потом → v0.8 (Whisper)**. Транскрипция голоса = огромный буст UX: клиенты любят отправлять голосовое, мастер не любит их слушать.

**Потом → v0.9 (надёжность)**.

**Когда мастеров станет больше одного → v1.0**.

## Технический долг (что я бы рефакторил независимо)

1. `bridge.py` — глобальное состояние, при перезапуске теряется. Перевести на in-memory dict с TTL или на колонку в `orders.master_active=true`.
2. Repository — функции, не класс. При росте проекта — обёрнуть в `OrdersRepo` класс с инжектом соединения.
3. `notify_master_new_order` возвращает кортеж — стоит сделать dataclass `NotifyResult`.
4. Хендлеры мастера толстые — выделить `services/order_service.py` для бизнес-логики (смена статуса, ответ клиенту).
5. Типы статусов `Literal["new", ...]` вместо `str` в сигнатурах.

## Принципы, которым следую

1. **MVP сначала, оптимизация потом** — каждый релиз должен закрывать боль, а не код-эстетику
2. **Простые решения** — SQLite пока хватает; PostgreSQL только когда упрёмся в реальные ограничения
3. **Без обратной совместимости hacks** — миграции и удаление старого кода вместо адаптеров
4. **Логи > молчание** — каждый сбой Telegram API логируется с контекстом
5. **Тред у мастера = одно окно на заявку** — это центральная UX-метрика проекта
